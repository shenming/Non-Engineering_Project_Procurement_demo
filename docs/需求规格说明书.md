### 2.2 详情主页面 (detail/index.html)

**功能标识：** F-PROC-DETAIL-001

**页面结构：**
- 顶部导航：返回列表 + 页面标题 + 面包屑
- 信息头卡片：项目名称 + 状态标签 + 操作按钮（编辑/删除/提交）
- Tab 页签：招采进度 / 审批流预览
- 招采进度页签：步骤进度条（circle-steps）+ 步骤内容 iframe
- 底部：上一步 / 下一步 / 取消

**步骤渲染逻辑（数据驱动）：**
- `init()` 从 `ctx.method` 读取采购方式 → 设置 `cfg = METHOD_CONFIG[method]`
- `selectedStep` 默认定位到第一个 `state === 'active'` 的步骤
- `renderSteps()` 根据每个步骤的 `state` 和选中态渲染 CSS 类名：

| 条件 | CSS 类 | 视觉 | 可点击 |
|------|--------|------|--------|
| `i === selectedStep` | `.selected` | 🔵 蓝底白字圆点 | ✅ |
| `state === 'done'` 且未选中 | `.done` | 🔵 蓝色边框白底 | ✅ |
| `state === 'active'` 且未选中 | `.done` | 🔵 蓝色边框白底 | ✅ |
| `state === 'future'` | `.future` | ⚪ 灰色 | ❌ |

- 步骤名从 `common/data.js` 的 `METHOD_STEPS` 自动读取，`detail/data.js` 的 `buildSteps()` 动态生成带状态的 steps 数组

**审批流预览（数据驱动）：**
- `renderApprovalFlow()` 从 `cfg.flow` 动态渲染表格
- 每行包含：节点描述、审批状态（tag标签）、审批日期、流程 ID

**节点加载：**
- `loadNode(index)` 将 `DETAIL_DATA` 存入 `sessionStorage`
- iframe 通过 URL 参数 `?method=&step=` 获取当前采购方式和步骤
- 节点页面从 `sessionStorage` 读取数据进行动态渲染

### 2.3 节点页面 (detail/nodes/*.html)

**数据来源：** 所有节点页面统一从 `sessionStorage` 读取 `procNodeData`

```javascript
var params = getUrlParams();
var method = params.method || '公开招标';
var step = parseInt(params.step) || 0;
var pd = JSON.parse(sessionStorage.getItem('procNodeData') || '{}');
var nodeData = pd[method].nodes[step];
```

## 3. 数据层架构

### 3.1 公共数据层 (`common/data.js`)

| 变量/函数 | 用途 |
|-----------|------|
| `CATEGORY_TREE` | 采购类别二级品类树 |
| `METHOD_STEPS` | 纯步骤名数组（无状态，新增/编辑/详情通用） |
| `getStepNames(method)` | 获取采购方式的步骤名数组 |

**5 种采购方式的步骤定义：**

| 采购方式 | 步骤数 | 步骤名 |
|----------|--------|--------|
| 公开招标 | 7 步 | 招采申请 → 招采公告 → 开标信息 → 上传评标结果 → 评标结果公示 → 中标结果 → 中标结果公示 |
| 邀请招标 | 5 步 | 招标申请 → 拟邀请单位 → 开标信息 → 上传评标结果 → 中标结果 |
| 询比采购 | 3 步 | 招标申请 → 拟邀请单位 → 中标结果 |
| 谈判采购 | 3 步 | 招标申请 → 拟邀请单位 → 中标结果 |
| 直接采购 | 2 步 | 招标申请 → 确定成交供应商 |

### 3.2 详情数据层 (`detail/data.js`)

| 变量/函数 | 用途 |
|-----------|------|
| `DETAIL_METHOD_CONFIG` | 详情配置（steps + files + flow），步骤名从 `METHOD_STEPS` 动态读取 |
| `METHOD_CONFIG` | 向下兼容别名，指向 `DETAIL_METHOD_CONFIG` |
| `buildSteps(method, states)` | 辅助函数，从 `METHOD_STEPS` 和 states 数组生成带状态的 steps |
| `getActiveStepName(method)` | 获取当前激活的步骤名 |
| `getDoneCount(method)` | 获取已完成步骤数 |
| `DETAIL_DATA` | 详情表单数据（info + nodes），供 iframe 节点页面使用 |

**详情配置数据（DETAIL_METHOD_CONFIG）：**

| 采购方式 | 步骤数 | 默认激活步骤 | files 映射 |
|----------|--------|-------------|------------|
| 公开招标 | 7 步 | 中标结果公示 | 7 个节点文件 |
| 邀请招标 | 5 步 | 开标信息 | 5 个节点文件 |
| 询比采购 | 3 步 | 拟邀请单位 | bid-apply → invite-units → award-result |
| 谈判采购 | 3 步 | 拟邀请单位 | bid-apply → invite-units → award-result |
| 直接采购 | 2 步 | 招标申请 | bid-apply → eval-result → award-result |
| 直接采购-小额 | 2 步 | 确定成交供应商 | bid-apply → eval-result |

### 3.3 数据流

```
common/data.js → METHOD_STEPS（步骤名唯一来源）
                    ↓
detail/data.js → buildSteps() 自动生成 { name, state }
                    ↓
         DETAIL_METHOD_CONFIG（steps + files + flow）
         DETAIL_DATA（表单数据）

list.html → sessionStorage(procurementViewContext)
  → detail/index.html init() 读取 ctx.method
    → 设置 cfg = METHOD_CONFIG[method]
      → loadNode() 存 DETAIL_DATA 到 sessionStorage
        → iframe 加载节点 → 从 sessionStorage 读取数据
          → 动态渲染表单 + 表格 + 附件
```
